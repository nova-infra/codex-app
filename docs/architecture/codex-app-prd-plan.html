02. 需求概述

2.1 需求简介

将 codex-app 从历史 TS/Bun workspace 迁移为 Go 主线运行时，并建立一套统一的 Codex 社交入口产品能力。核心不是“换语言”，而是把 project、provider、codex_home、session、command、approval、render、channel readiness 都收敛到 Go runtime 的稳定 contract。

2.2 业务目标

  • 让用户可以在 Telegram、Weixin、Lark/Feishu 中稳定触发 Codex turn，并获得符合平台能力的低噪音回复。
  • 让开发者只维护一套 Go runtime 与一套 render event contract，不再在各 channel 里重复写进度、审批、媒体、错误和最终回复逻辑。
  • 让部署方可以通过配置检查、凭据预检、dry-run 和最终装配输出判断当前实例是否可启动、可连通、可回滚。

2.3 预期收益

一致性provider、model、mode、cwd、session、approval 命令跨 channel 表现一致。
低噪音reasoning 默认隐藏，工具进度按平台能力展示，最终答案不与进度混杂。
可迁移TS workspace 在 Go 端端到端可用后退出主线，保留迁移资料而不继续扩功能。

03. 目标与非目标

本期目标
  • Go CLI 可启动、可 doctor、可 dry-run、可 render-demo。
  • 配置模型支持 project、provider refs、project-local CODEX_HOME。
  • Codex bridge 支持 app-server launch command、session resume/fallback、provider config 写入。
  • 统一 render event 与 DisplayProfile：high / medium / low / minimal。
  • TG、Weixin、Lark 三端至少各完成一条真实消息链路。
本期非目标
  • 不扩成 cc-connect 的多 Agent 平台。
  • 不恢复 Web channel 为主线入口。
  • 不建设 Web 管理后台。
  • 不把 cron、relay、hooks、账号矩阵纳入第一轮。
  • 不让 channel 拥有 session、approval、provider、tool state 的真相源。

04. 用户与场景

角色核心诉求典型场景
社交入口用户在熟悉的聊天工具里调用 Codex，少看到无意义过程噪音。在 TG 发起仓库问题；在 Weixin 发图并要求处理；在 Lark 群里 @ 机器人获取进度和结果。
部署/运营者快速知道服务能不能启动、哪个 channel 缺凭据、是否会污染主 Codex Home。运行 doctor、serve --dry-run、查看 readiness 与配置渲染结果。
开发者新增或修改 channel 时不用复制 Codex 事件处理和渲染逻辑。修改 internal/render 一处规则后，TG/Weixin/Lark 均按 profile 生效。
维护者从 TS 主线平滑迁移到 Go，并在真实链路可用后删除 legacy。先保留 TS 作为迁移资料，Go 三端通过验收后清理 packages/ 与 Bun 配置。

核心使用流

1用户在 channel 发送消息或命令
2channel runtime 归一输入和上下文
3core 校验 project、provider、session、permission
4Codex bridge 启动或恢复 app-server session
5render pipeline 过滤、分段、提取媒体、生成 RenderEvent
6platform renderer 按 DisplayProfile 输出结果

07. 功能需求

FR-1 Go Runtime 基线

项要求
前置条件仓库已存在 go.mod 与 cmd/codex-app；TS/Bun 文件可作为迁移资料，但不再新增功能。
成功路径用户运行 go run ./cmd/codex-app --help、doctor、serve --dry-run 可得到稳定输出。
失败路径配置缺失、provider 无效、channel 凭据缺失时 fail fast，并指出缺失项和修复方向。
验收标准CLI help 可用；doctor 能校验 project/provider/codex_home；dry-run 不启动外部连接也能输出装配结果。

FR-2 Project / Provider / CODEX_HOME

规则说明
配置真相源project 是 channel 绑定和 Codex 执行上下文的根；provider 只能通过 ref 引用，不由 channel 直接拼接。
隔离规则每个 project 默认使用 project-local CODEX_HOME，避免污染用户主 ~/.codex。
写入规则Codex bridge 负责生成 provider config payload；写入前应可 dry-run 展示。
异常规则provider ref 不存在、模型为空、codex_home 不可写、cwd 不存在时阻止启动。

FR-3 Session 与 Command Router

能力要求验收
Session ownershipsession 真相源由 core/session 管理，channel 只传入 user、thread、project 与消息。关闭任一 channel 不影响已有 session 存储结构。
Resume / fallback有 session id 时尝试恢复；恢复失败时按规则创建新 session，并把 fallback 原因写入事件。session store 单测覆盖恢复、缺失、损坏、fallback。
Command Routerprovider、model、mode、cwd、session、approval 命令走统一 router。TG/Weixin/Lark 命令语义一致，平台只负责输入输出形式。
Approvalapproval 事件由 core 生成，renderer 按平台能力输出 inline button、数字菜单或 card button。三个 channel 均能完成 confirm/reject，并能处理超时或无效输入。

FR-4 Hermes Render Pipeline

NormalizeCodex raw event 转 RenderEvent
Filter默认过滤 reasoning 与 think 标签
Segment工具事件切断 assistant stream
Extract媒体 URL、文件、图片、音频从正文剥离
Profile按 DisplayProfile 选择展示策略
Fallback平台能力失败时降级为安全文本
平台Profile展示规则失败处理
Telegramhigh允许 stream/edit，显示完整工具进度，使用 inline keyboard。edit 失败则发新消息；HTML 失败则转纯文本。
Weixinlow只做 typing、最终纯文本、数字审批、媒体 relay；不展示 reasoning 和工具进度。长文本分块；媒体失败时给出文本说明。
Lark/Feishumedium卡片优先，只显示新阶段，按钮审批，保留群聊 thread/reply 上下文。卡片失败 fallback text；不能持续新增重复进度卡。

FR-5 Channel Runtime

Channel本期交付不做
Telegram文本、图片/文件、审批按钮、session 命令、provider/model 命令、最终回复。不在 TG 内部维护独立 session 和 provider 规则。
WeixiniLink 收发、数字审批、文本分块、图片 relay、typing。不复制 TG 的工具进度流和 edit 逻辑。
Lark/Feishu@ 触发、卡片审批、fallback text、群聊 reply/thread 上下文、WS readiness。不进入文档/表格/复杂审批套件。

8.1 核心字段

字段中文名类型来源校验规则

8.2 状态与异常

状态进入条件退出条件异常处理
ConfigInvalid配置缺失或字段不合法。用户修正配置并重新 doctor。阻止启动，输出具体字段路径。
ChannelNotReadychannel 启用但凭据、webhook、WS 或权限缺失。凭据补齐并 readiness 通过。dry-run 可展示，正式 serve fail fast。
SessionStarting收到用户消息，需要新建或恢复 Codex session。app-server session 可用。恢复失败时 fallback，新建原因进入 render event。
TurnStreamingCodex 产生 delta 或 tool event。收到 final 或 error。renderer 按 profile 合并、编辑、分块或隐藏。
ApprovalWaitingCodex 请求用户确认。用户确认/拒绝/超时。无效输入提示一次；超时按 reject 或 expired 规则处理。
DeliveryFallback卡片、edit、媒体或富文本发送失败。fallback text 发送成功或记录最终失败。不能丢失最终答案；失败原因进入日志。

09. 指标与验收

9.1 关键指标

指标口径目标
doctor 通过率启用 channel 的配置和凭据检查通过数 / 启用 channel 数。上线前 100%。
真实消息链路成功率手机端或群聊发起 Codex turn，收到最终回复的成功率。TG/Weixin/Lark 各至少 1 条通过，灰度期持续记录。
render fallback 率富文本/card/edit/media 失败后 fallback text 的次数。可观测；高于阈值时回退平台策略。
session resume 成功率带 session id 请求成功恢复的比例。灰度期建立基线。
噪音事件曝光reasoning、内部 tool item、无意义 started 事件被用户看到的次数。默认 0。

9.2 阶段验收

Phase 0冻结 TS 主线清点未提交改动；TS/Bun 仅作为迁移资料，不再加功能。
Phase 1Go 骨架go run ./cmd/codex-app --help 可用；默认配置路径可打印。
Phase 2配置与 project/providerdoctor --config 和 serve --dry-run --config 能验证 provider/project/codex_home。
Phase 3Core enginesession store、channel credential preflight、Hermes render contract 均有 Go 测试。
Phase 4Codex agent生成 app-server launch command，并进入真实 session resume/fallback。
Phase 5Telegram E2ETG 手机端完成一条真实 Codex turn，含审批或工具事件。
Phase 6Weixin E2EWeixin 完成真实 Codex turn，展示 low tier 低噪音结果。

11. 发布计划

11.1 发布方式

  1. 先以 Go CLI 本地可运行为基线，验证 doctor、serve --dry-run 和 render demo。
  2. 先接 Telegram 作为完整样板，再接 Weixin 低噪音链路，最后接 Lark 卡片链路。
  3. 三端真实链路通过后，再删除 TS/Bun legacy 文件和发布流程残留。

11.2 灰度策略

阶段范围退出条件
Alpha本地 CLI + dry-run + 单元测试。配置、session、render、channel readiness 测试通过。
Beta单项目单 channel，优先 Telegram。真实 turn、approval、错误 fallback 均通过。
Gamma多 channel 同项目，加入 Weixin 与 Lark。跨 channel 命令语义一致，session 不串线。
ReleaseGo 主线替代 TS 主线。legacy 删除后测试和文档不再引用 Bun/TS 主线。

11.3 回滚策略

  • 在 Telegram E2E 通过前不删除 TS workspace。
  • 在三端 E2E 全部通过前不把 TS legacy 从 git 历史之外完全不可达化。
  • provider config 写入必须具备 dry-run 输出；正式写入前记录目标路径和变更摘要。
  • 单个 channel readiness 失败时允许先下线该 channel，但不能影响 core 配置诊断。

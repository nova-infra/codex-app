01. 一句话结论

建议重写而不是继续修补 TypeScript workspace。 当前目标已经从轻量 Bun 网关转向 cc-connect 风格的长期运行时，核心复杂度在 project/provider/session/channel engine，而这些能力 cc-connect 的 Go 实现已经更接近目标。
保留 codex-app 定位仍然是 Codex 专用社交入口，不扩成 cc-connect 的全 Agent 平台。
复用 cc-connect 主线重点复用 Codex app-server、provider config、project model、platform optional interfaces。
先打通一个入口优先 Telegram 端到端可用，再迁 Weixin，最后 Feishu/Lark。

07. 阶段计划

Phase 0

冻结 TS 主线

清点当前 Bun/TS 未提交改动，决定保留为迁移资料还是丢弃。标记 packages/* 为 legacy，不再加功能。

Phase 1

初始化 Go 骨架

创建 go.mod、cmd/codex-app、config、core、agent/codex、platform、daemon。先保证 go run ./cmd/codex-app --help 可用。

Phase 2

迁移配置与 project/provider 模型

从 cc-connect 裁剪 config，支持多 project、provider refs、project-local codex_home。

Phase 3

迁移 core engine

迁移 platform interface、message handler、session、command router、project state。所有 slash command 收敛到 core。

Phase 4

迁移 Codex agent

打通 app-server session、provider config 写入、session resume/fallback、usage/context 读取。

Phase 5

打通 Telegram

作为第一个端到端入口，覆盖绑定、命令、审批、回复、stream/edit 展示。

Phase 6

打通 Weixin

迁移 iLink、数字审批、纯文本分块、图片输入输出。

Phase 7

打通 Feishu / Lark

迁移卡片、群聊 @、fallback text、thread/reply context。

Phase 8

删除 Bun workspace

确认 Go 版可用后删除 packages/、package.json、bun.lock、tsconfig.json、.changeset/，更新 README 和脚本。

08. 命令与验证

开发期命令

go test ./...
go vet ./...
go run ./cmd/codex-app --help
go run ./cmd/codex-app doctor
go run ./cmd/codex-app doctor --config ./config.json
go run ./cmd/codex-app serve --dry-run
go run ./cmd/codex-app serve --dry-run --config ./config.json
go run ./cmd/codex-app render-demo --channel all
go run ./cmd/codex-app capabilities list --channel all
当前进度：Go 入口已具备 JSON 配置加载、provider/project 校验、project-local CODEX_HOME payload、Codex launch command skeleton、文件 session store skeleton、三端 channel credential preflight，以及 doctor/serve --dry-run/render-demo smoke。非 dry-run 会在缺少 Telegram/Weixin/Lark 凭据时 fail fast，不再使用 10ms placeholder。

每阶段最低验收

  • Phase 1：CLI 能启动，默认配置路径可打印。
  • Phase 2：doctor --config 和 serve --dry-run --config 能验证 provider/project/codex_home。
  • Phase 3：session store、channel credential preflight、Hermes-style render contract 均可通过 Go 测试验证。
  • Phase 4：能生成 Codex app-server launch command，并进入真实 session resume/fallback 实现。
  • Phase 5：Telegram 手机端完成一次真实 Codex turn。
  • Phase 6：Weixin 手机端完成一次真实 Codex turn。
  • Phase 7：Feishu/Lark 群聊或私聊完成一次真实 Codex turn。
  • Phase 8：rg "bun|packages/" 不再命中 active 主线。

09. 风险与处理

风险处理方式
照搬 cc-connect 导致范围失控只迁 Codex 主线和三个社交平台；其他 Agent/platform 不进第一轮。
Go 重写期间功能断档Telegram 端到端跑通前，不删除 TS workspace。
污染用户主 Codex 状态默认 ~/.codex-app/codex-home，配置中显式声明。
channel 返回样式再次分叉core 中定义 display profile，platform 只消费，不自造格式。
Feishu/Lark 提前复杂化先做文本、卡片审批、fallback；文档/多维表格后置。

10. 最终验收口径

  1. go test ./... 通过。
  2. go run ./cmd/codex-app doctor 通过。
  3. Telegram / Weixin / Feishu-Lark 至少各完成一条真实手机端消息链路。
  4. provider、model、mode、cwd、session 命令跨 channel 表现一致。
  5. 默认状态不读取或写入用户主 ~/.codex。
  6. Bun workspace 已退出 active 主线。
  7. Web channel 没有回到主规划。

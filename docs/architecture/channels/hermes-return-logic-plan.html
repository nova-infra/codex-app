深化 Hermes 返回逻辑

本方案把 /Users/Bigo/Desktop/develop/ai/hermes-agent 的 Gateway 展示策略整理成 codex-app 的 channel renderer 规范：统一隐藏 reasoning、按平台能力选择 tool progress、流式编辑、卡片、纯文本降级和媒体回传。

核心原则最终回答与工具进度分层，不让内部过程污染正文。
TGhigh tier：可展示完整进度，但必须编辑复用。
Weixinlow tier：不刷进度，只发最终纯文本和必要附件。
Feishu/Larkmedium tier：卡片优先，只显示新阶段，失败文本 fallback。

01. 结论

建议不要在 TG / Weixin / Feishu 各自硬写返回格式。 应抽一个 Hermes-style renderer：core 产生统一事件，renderer 根据平台 profile 决定哪些事件可见、如何合并、如何编辑、如何 fallback。
问题现状目标
reasoning 泄漏TG/Weixin 代码里仍有 thinking 状态和 reasoning delta 路径。默认全部隐藏；只保留显式 debug 开关。
工具进度噪音Weixin 会把 item/started 发成永久文本；TG 也存在进度与最终答案顺序问题。TG 编辑同一条；Weixin 默认不发；Feishu 只发新阶段。
流式与工具交错文本 delta、tool started、final answer 容易乱序。工具边界强制分段：progress bubble 在上，final answer 在下。
媒体标记外泄图片/文件内部字段可能被当文本处理。MEDIA:、音频占位、内部 JSON 全部在 renderer 前剥离，附件走 channel capability。

02. Hermes 返回模型

Hermes 的关键不是 UI 好看，而是把“用户该看到什么”变成平台能力的函数。参考点：

  • gateway/display_config.py：按平台解析 tool_progress / show_reasoning / streaming / cleanup_progress。
  • gateway/stream_consumer.py：流式编辑、工具边界分段、thinking 标签过滤、长文本拆分。
  • gateway/platforms/helpers.py：文本批处理、Markdown 降级、去重。
  • gateway/platforms/feishu.py：卡片/富文本 fallback、反应状态、审批按钮。

03. 事件流水线应该这样改

Step 1

统一事件归一

先在 core 层把 Codex 事件归一为 assistant.delta、tool.started、tool.completed、approval.requested、turn.completed、error。平台层不要直接判断 Codex 原始 method。

Step 2

默认过滤 reasoning

所有 reasoning item、summaryTextDelta、<think> 标签在 renderer 前被过滤。只有显式 display.show_reasoning = true 才允许展示。

Step 3

工具边界分段

收到 tool started/completed 后，当前 assistant stream 要结束当前段，后续正文必须新开消息或新内容块，避免“答案写到工具进度上方”。

Step 4

媒体剥离

在文本渲染前扫描 MEDIA:、图片 URL、文件路径、音频占位；附件走 ImageSender/FileSender，正文只保留必要说明。

Step 5

平台 profile 决策

renderer 根据 toolProgress、streaming、messageUpdate、card、plainText 决定最终展示方式。

04. 平台 Display Profile

type DisplayProfile struct {
    ToolProgress      string // all | new | off
    ShowReasoning     bool
    Streaming         bool
    MessageUpdate     bool
    ToolPreviewLength int
    CleanupProgress   bool
    Format            string // html | markdown | card | plain
}

TelegramhightoolProgress=all、streaming=true、messageUpdate=true、format=html支持 editMessageText 和 inline keyboard，适合展示完整过程。
WeixinlowtoolProgress=off、streaming=false、messageUpdate=false、format=plainiLink 文本消息是永久消息，刷进度会污染聊天。
Feishu/LarkmediumtoolProgress=new、streaming=true、messageUpdate=true、format=card团队群聊需要少噪音，卡片能承载状态和审批。

05. Telegram 应该如何修改

当前要收敛的问题仍存在 classic/hermes 双模式、thinking map、progress/final 顺序处理分散在 adapter/notifications/streamEditor。
目标TG 平台只负责 Bot API、HTML 降级、inline button、message update；显示策略来自 DisplayProfile。

修改建议

  1. 删除默认 classic 分支，默认只保留 Hermes-style；classic 如要保留只能作为 debug profile。
  2. item/reasoning/summaryTextDelta 默认直接丢弃，移除用户可见 thinking line。
  3. tool progress 只维护一条 progress message，收到新 tool 时编辑同一条；不要每个工具发新消息。
  4. tool boundary 后调用 stream segment break，让最终答案在 progress 后作为新消息出现。
  5. 最终答案使用 HTML 优先；失败 fallback plain text；长文本按 Telegram 限制拆分。
  6. approval 继续使用 inline keyboard，但文案走统一 approval renderer。
  7. 可选支持 cleanupProgress=true：成功后删除或折叠进度，失败保留错误线索。

2. ...

不要出现：<think>、Thinking...、连续 10 条工具状态、最终回答插在工具消息上方。

06. Weixin 应该如何修改

当前要收敛的问题Weixin 目前会处理 item/started 并调用 sendProgress，还会 append thinking；这与 Hermes low tier 不一致。
目标Weixin 只做 typing、最终纯文本、数字审批、图片/文件 relay；不展示工具进度和 reasoning。

修改建议

  1. item/started 对非图片类工具默认不发送任何文本；只保留内部状态或日志。
  2. item/reasoning/summaryTextDelta 默认丢弃，删除 thinkingByChatId 用户可见路径。
  3. 保留 typing indicator，作为“正在处理”的唯一实时反馈。
  4. 最终回复调用 Markdown-to-plain formatter，保留代码缩进、链接语义和段落。
  5. 长文本分块时第一条可以加摘要，但不要发“详细内容”这类固定话术噪音过多。
  6. 图片生成/查看结果只走图片 relay；正文不暴露内部 URL / JSON / CDN 字段。
  7. approval 保持数字菜单：回复 1 确认，2 拒绝。

07. Feishu / Lark 应该如何修改

当前状态codex-app 里 Feishu/Lark 仍是规划项；Go 重写时应直接按 Hermes medium tier 落地，不走 TG/Weixin 的旧逻辑。
目标卡片优先、文本 fallback、只显示新阶段、隐藏 reasoning、群聊保持 reply/thread 上下文。

修改建议

  1. 新增 platform/feishu，按本仓库 channel contract 实现 card、preview cleaner 设计。
  2. tool progress 使用 new：同类重复工具不刷，只在阶段变化时更新卡片。
  3. 进度卡片可编辑则编辑；编辑失败再发一条 fallback 文本，不能持续新增卡片。
  4. approval 使用交互卡片按钮：确认 / 拒绝；卡片失败 fallback 为文本按钮说明。
  5. 群聊中必须解析 @ 机器人；私聊直接触发。
  6. 保留 thread/reply metadata，但不要把 Feishu thread 结构泄漏到 core。
  7. 用 reaction 或 card 状态表达“处理中/失败”；成功不强制发额外 success 噪音。

09. 迁移顺序

P0

先抽 renderer profile

先在 Go core 定义 profile，不急着接三个平台。profile 是后续一致性的根。

P1

Telegram 先落地

因为 TG 支持 edit 和 inline keyboard，最适合作为完整 renderer 的第一条端到端样板。

P2

Weixin 只做低噪音

迁 Weixin 时不要复制 TG 进度逻辑，直接按 low tier：typing + final + approval + media。

P3

Feishu/Lark 做卡片策略

最后接 Feishu/Lark，重点做卡片 fallback 和群聊 thread/reply 上下文。

10. 验收口径

验收项通过标准
reasoning 默认隐藏三端真实消息中不出现 thinking 标签、reasoning summary、scratchpad。
TG 高信息密度能看到必要工具进度，但通过编辑复用，不刷屏。
Weixin 低噪音工具执行期间只有 typing；完成后只发最终回复和必要附件。
Feishu 卡片 fallback卡片可用时卡片展示；卡片失败时文本仍可完成审批和最终回复。
媒体不外泄用户看不到 MEDIA:、内部 URL JSON、临时文件路径。
顺序稳定tool progress 永远在最终答案之前，最终答案不被插入到工具消息中间。
执行提醒：这份文档是修改建议，不是代码实现。后续真正实现时应先按 Go 重写计划落 core renderer，再接 TG / Weixin / Feishu。
